## 2-1. 不具合

バグはあったか：

あった場合：

原因（選択）：
- [ ] 設計ミス
- [x] 実装ミス
- [ ] テスト不足
- [x] その他: (MacとWindowsの環境差異)

具体的に：


- `apps/api/.env` の `TIDB_SSL_CA=/etc/ssl/cert.pem` が Mac/Linux 専用のパスだったため、Windows 環境では SSL 証明書が見つからず TiDB への接続が失敗した（`Connections using insecure transport are prohibited.`）
→ 開発・テストが Mac 中心で進んでおり、Windows での動作確認がなかった
→ `.env.example` に OS ごとのパス設定手順を記載することで対応（[#131](https://github.com/jyogi-web/Jyogi_Navi/issues/131)）


- Discord メッセージを TiDB に保存する際、全メッセージを1レコードの `content` カラムに結合して保存していたため、検索・RAG に使えないデータになっていた（[#130](https://github.com/jyogi-web/Jyogi_Navi/issues/130)）
→ `normalize.py` の実装ミスで、1メッセージ = 1レコードの構造になっていなかった
→ `to_faq_records()` を追加し、著者・チャンネル・タイムスタンプを `metadata` に格納する形に修正（[#139](https://github.com/jyogi-web/Jyogi_Navi/pull/139)）

### OK例
- APIのレスポンス形式を途中で変更し、フロント側が追従していなかった
→ API契約を設計で固定していなかった

# 5. 技術・アーキテクチャ

## 5-1. 技術選定

- なぜ選んだか：
  - **Next.js（App Router）**：SSR/ISRによる高速化と、Cloudflare Workers（OpenNext）へのデプロイが無料枠内で完結するため
  - **FastAPI（Python）**：RAGエコシステム・Difyとの整合性が高く、自動ドキュメント生成（OpenAPI）がそのまま使えるため
  - **Dify（セルフホスト）**：RAGパイプラインをGUIで管理でき、Chat APIとしてそのまま公開可能。クラウド版の無料枠では要件を満たせず、コスト0円を優先してセルフホストを選択
  - **TiDB Serverless**：ベクトル検索＋全文検索のハイブリッドに対応し、12GB無料枠が圧倒的。Difyが標準対応しているため
  - **Supabase（PostgreSQL）**：Difyの推奨内部DB。MySQL系でエラー事例があったため
  - **Cloudflare Pages / Tunnel**：フロントエンドのCDN・DDoS防御が標準付属。Tunnelで自宅PCのDifyをポート開放なしでHTTPS公開できるため

- 結果どうだったか：
  - Next.js + Cloudflare Pages：問題なく動作。OpenNextによる対応も想定通り
  - FastAPI + OpenAPI：スキーマ駆動開発が機能し、FE/BEのズレを防げた
  - **Dify セルフホスト：最大の誤算**。セットアップ難易度を過小評価しており、立ち上がるまでの間チーム全体の作業がブロックされた。コスト優先の判断自体は妥当だったが、代替手段を用意すべきだった
  - TiDB：接続時のSSL証明書パス（Mac/Windows差異）でハマったが、設定変更で解決
  - Supabase：Dify専用として安定動作。問題なし

---

## 5-2. 拡張性

- 機能追加しやすいか：しやすい。OpenAPI スキーマ駆動でフロント型が自動生成されるため API 追加時の FE/BE のズレが起きにくい。Dify の RAG パイプラインは GUI で変更できるためプロンプト調整・LLM 切り替えはコード不要

- どこが詰まりそうか：
  - Dify が自宅 PC 依存のため、利用者増加時にスペック不足でスケールアップが難しい
  - Embedding・ベクトル検索が未完成のため、RAG 精度の改善は今後の実装次第で品質が不安定
  - チャット入力の PII マスキングが未実装のまま機能追加するとセキュリティリスクが広がる

# 6. 運用

## 6-1. 運用できるか

- 実際に回るか：Difyが自前で構えられれば回る、GeminiAPIのコスト面は課題
- 手動作業はどれくらい：

  | 作業 | 頻度 | 担当依存 |
  |---|---|---|
  | Dify セットアップ・docker-compose up | 初回 + 障害時 | 担当者のみ（手順書なし） |
  | Cloudflare Tunnel の再起動（PCリブート後） | 不定期 | 担当者のみ |
  | APIキー更新（Gemini / Discord / TiDB） | 失効時 | 要対応 |
  | Terraform 初回 apply | 初回のみ | CI外 |
  | OpenAPI 生成・クライアントコード生成 | API変更時（CIで強制） | 自動化済み |

  → 自動化できていない作業がDify・Tunnel周りに集中しており、担当者依存が高い状態


---

## 6-2. 障害対応

ログで原因分かるか：

- バックエンドのログの構造化
  - GoogleCloudLogで見られるようにしている
  - APIリクエストのログにリクエストIDを付与しているため、フロントエンドのエラーと紐づけて原因分析ができる

- 復旧手順あるか：一部はある
  - Cloud Run・TiDB・Supabase はマネージドサービスのため自動復旧
  - Dify（自宅 PC）は `docker-compose up -d` で再起動できるが手順書がなく担当者依存。
    Cloudflare Tunnel も同様に明文化されていない

## 7-2. 使い方（使いまわせる機能ごとに1つずつ）

### OpenAPI スキーマ駆動開発（型自動生成）
- セットアップ：`pnpm add -D @hey-api/openapi-ts` を追加し、`openapi-ts.config.ts` に生成設定を記述。CI で未生成の場合エラーになるよう `package.json` の scripts に組み込む
- 利用方法：FastAPI 起動後 `pnpm generate` を実行すると `src/client/` に TypeScript SDK・Zod スキーマ・TanStack Query フックが自動生成される

### データクリーニング正規表現ルール
- セットアップ：`scripts/ingest/*/normalize.py` をコピーし、対象サービスに合わせてパターンを修正
- 利用方法：テキストを `normalize()` に渡すと Discord UID・メールアドレス・URL の PII が除去されたテキストが返る

### 開発環境統一（lefthook / CI 設定）
- セットアップ：`lefthook.yml` と GitHub Actions ワークフローファイルをコピーし、プロジェクトに合わせてコマンドを修正
- 利用方法：`git commit` 時に自動でリント・フォーマットが走る。CI で型チェック・テストが自動実行される
